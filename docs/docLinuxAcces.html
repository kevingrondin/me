Accés à distance

Un peu loin dans la configuration d'accés à distance

Clée SSH

Definition

la clée privé vous appartient, seul la clée public doits être sur les différents serveur

Générés les clées sous Linux

# Version secure
ssh-keygen -t rsa -b 4096 -C "email_address"

# Version basic
ssh-keygen -t rsa -C "email_address"

Rendre accéssible c'est fichier générés par le systèmes

sudo chmod 600 ~/.ssh/id_rsa
sudo chmod 600 ~/.ssh/id_rsa.pub

Github aussi peut fonctionner avec vos clée SSH, rajouter la clée Publique dans Settings > SSH and GPG keys

ssh -T git@github.com

Maintenant plus besoin de s'authentifier depuis cette machine pour vos projet github ^^

Générés les clées sous Windows

même ligne que sous Linux, depuis votre powershell et ensuite vous pouvez afficher votre clée publique avec la commande type

type C:\Users\name\.ssh\id_rsa.pub

Mettre la clée sur le rasp

Avec cette ligne vous aller envoyer votre clée publique, grace au username et ip de votre rasp, dans le fichier authorized_keys de celui-ci

cat ~/.ssh/id_rsa.pub | ssh <USERNAME>@<IP> 'mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys'

marche aussi avec rsyn exempel ci-dessous

rsync -e « ssh -i /root/.ssh/<KEY> » -av <USERNAME>@<IP_ADRESSE>:/source/destination/

Autoriser les clées

Autoriser les authorized_keys

nano /etc/ssh/sshd_config

et decommentez

AuthorizedKeysFile   /.ssh/authorized_keys

et recharger la config ssh

/etc/init.d/ssh force-reload

Connexion sans mot de passe

vous pouvez vous connectez avec cette ligne de commande depuis la machine ou il y'a votre clée privée

ssh -i /root/.ssh/<KEY> <USERNAME>@<IP>

WSL sous windows

inutile de copier la clée privée de votre windows dans votre wsl, pour se connecter au rasp

Avec Pageant et wsl-ssh-pageant-amd64-gui votre clée privée et aussi importer dans votre WSL

/!\ avec WSL 2 ça peut changer il n'y aura peut-être plus besoin de faire cette manipulation

Astuce

Configurer les HOST pour une connexion encore plus rapide

Dans votre dossier .ssh creer un fichier config sans extension, créer des racourcis comme ci-dessous

/!\ j'ai mis en commantaire un tunnel proxy car pour l'instant y'a pas besoin ^^

Host *
   ForwardAgent yes

Host <ALIAS>
   HostName <SERVEUR>
   Port     <PORT>
   User     <USER>
   # Tunnel SSH
   # ProxyCommand     ssh <USER>@<SERVEUR> -W %h:%p

Sous linux même chose mais, avec les autorisation systèmes

nano ~/.ssh/config
chmod 600 ~/.ssh/config
chown $USER ~/.ssh/config

sudo apt install -y xrdp

Samba de base

sudo apt install -y samba

configurer samba

mkdir /home/pi/shares
chmod 777 /home/shares/public

sudo rm /etc/samba/smb.conf
sudo nano /etc/samba/smb.conf

[global]
        netbios name = server
        server string = server
        workgroup = WORKGROUP
        security = user

[server]
        comment = server
        path = "/home/pi/shares"
        public = yes
        guest ok = yes
        read only = no

sudo pdbedit -a -u pi
sudo service smbd restart

Samba avancé

On créer un groupe

groupadd <group>

On créer un dans le groupe 

useradd -g <group> <user>

On lui donne un mot de passe pour samba

smbpasswd -a <user>

On créer un dossier partagé avec les droit adéquat

mkdir -p /<partage>/<groupe>
chown -R root:<groupe> /<partage>/<groupe>/
chmod -R 770 /<partage>/<groupe>/

On historise la configuration par defaut de samba

mv /etc/samba/smb.conf /etc/samba/smb.conf.bak

vi /etc/samba/smb.conf
 
# Ajouter la configuration ci-dessous
 
#======================= Global Settings =======================
 
[global]
workgroup = WORKGROUP
server string = %h
public = yes
security = user
encrypt passwords = true

#======================= Share Definitions =======================

[<group>]
comment = répertoire du <groupe>
  path = <partage>/<groupe>
  valid users = @<groupe>
  writable = yes
  directory mask = 0770
  create mask = 0770
  force create mode = 0770

dans valid users on écri @groupe1, @groupe2 et user1, user2

On test si tout es bon

testparm smb.conf
